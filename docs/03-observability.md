# Observability

## Overview

The Anomaly Inference System is instrumented using Prometheus and Grafana to provide end-to-end visibility across the API layer, asynchronous queue, worker processing, AI inference, and Kubernetes infrastructure.

The observability strategy is designed to answer five operational questions:

- Is the platform healthy?
- Where is latency introduced?
- Which component is the current bottleneck?
- Is the autoscaler making the correct scaling decisions?
- Is the candidate model behaving differently from the production model?

Each layer exports metrics independently, allowing performance bottlenecks to be isolated through latency decomposition.

# Triton Client Metrics

The worker records client-side latency while communicating with Triton.

These metrics complement Triton's built-in server metrics.

Metrics

- `worker_triton_request_latency_seconds`
- `worker_triton_serialization_seconds`
- `worker_triton_deserialization_seconds`

Together with Triton's server-side latency metrics, these measurements provide complete end-to-end latency decomposition.

# Model Behaviour Metrics

During shadow traffic validation, the platform compares production and candidate model behaviour.

## Prediction Agreement

Measures whether both model versions produce the same classification.

Metrics

- `prediction_agreement_total`
- `prediction_disagreement_total`

## Prediction Distribution

Tracks the number of anomaly and normal predictions generated by each model version.

Metric

- `prediction_result_total`

This enables comparison of anomaly rates between model versions during shadow validation.


## Score Distribution

Tracks anomaly score distributions for each model version.

Metric

- `prediction_score`

This helps detect subtle behavioural differences before prediction disagreement becomes significant.

## Score Difference

Measures the absolute score difference between the stable and candidate models.

Metric

- `prediction_score_diff`

This provides an early signal of model drift even when both models continue producing identical classifications.

# Observability Philosophy

The platform follows latency decomposition rather than relying solely on end-to-end request latency.

Performance is analysed across multiple stages:

```text
Client Request
        │
FastAPI
        │
Redis Queue Wait
        │
Worker Processing
        │
Serialization
        │
Worker → Triton Network
        │
Triton Queue
        │
Compute Input
        │
Model Inference
        │
Compute Output
        │
Deserialization
        │
Client Response
```

This approach enables rapid identification of infrastructure bottlenecks, inference bottlenecks, and model rollout regressions while supporting production operations and incident response.
